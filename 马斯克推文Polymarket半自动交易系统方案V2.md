# 马斯克推文 Polymarket 半自动量化交易系统设计方案 V2

> 版本定位：从“自动下单交易 Bot”升级为“半自动可视化交易驾驶舱 + 概率研究系统”。  
> 核心目标：先验证策略是否真的有正期望，再逐步从观察、纸面交易、半自动实盘，演进到有限自动化交易。

---

## 一、版本更新说明

本版本基于原方案和后续讨论，对策略逻辑做了几个关键修正：

1. 不再把“买入 2-3 个区间且价格和小于 100%”直接定义为套利。
2. 将策略核心改为：
   - 主策略：高概率区间篮子的概率交易。
   - 副策略：开盘或低估阶段的价格收敛交易。
3. 入场条件从“区间价格之和低于某阈值”升级为“模型概率、可成交成本、滑点、手续费、安全边际”的综合判断。
4. 推文计数以 xtracker 的结算口径为核心，其他数据源作为补充和校准。
5. 速率模型从简单线性外推升级为三层模型：历史基础概率、当前进度修正、异常风险识别。
6. 止损从单一价格止损升级为价格、速率、流动性、时间、数据异常和组合风险的多维风控。
7. 产品形态优先做半自动可视化页面，先让人能观察、确认和复盘，再逐步自动化。

---

## 二、系统目标

### 2.1 新目标

构建一个基于 xtracker 结算口径、Polymarket 盘口数据和历史概率模型的半自动可视化交易系统。

系统需要完成：

1. 自动发现 Polymarket 上活跃的 Elon tweets 市场。
2. 自动拉取每个区间 outcome 的盘口、价格、成交量和流动性。
3. 自动拉取 xtracker 当前 Post Counter，并尽量保持与结算口径一致。
4. 基于历史数据和当前进度，估计每个区间的命中概率。
5. 计算市场价格与模型概率之间的偏差，寻找正期望或价格收敛机会。
6. 在可视化页面中展示所有数据、模型判断、推荐篮子、风险状态和操作建议。
7. 初期只做观察和纸面交易，中期支持人工确认下单，后期再有限自动化。

### 2.2 策略定位

本系统不是单纯寻找无风险套利，而是寻找以下两类机会：

```text
机会一：模型认为某组区间的真实命中概率高于可成交总成本。
机会二：开盘或市场低估阶段，高概率区间篮子价格偏低，后续有价格收敛空间。
```

也就是说，系统赚的不是“无条件的 1 - 买入成本”，而是在命中概率、市场定价、时间进度和交易流动性之间寻找优势。

## 三、核心策略框架

### 3.1 原策略需要修正的地方

原方案中的核心想法是：

```text
买入 2-3 个高概率区间。
如果这些区间 Yes 价格之和低于 100%，命中后赚取差价。
```

这个方向可以保留，但需要重新定义。

只有当买入了完整、互斥、穷尽的所有 outcome，且总成本小于 1 时，才是近似无风险套利。

如果只买 2-3 个区间，实际盈亏是：

```text
命中所选区间之一：获得 1 USDC payout。
没有命中所选区间：这些 Yes 仓位归零。
```

因此，买入部分区间的篮子交易，其盈亏平衡条件是：

```text
所选区间的真实覆盖概率 > 买入这些区间的实际成本。
```

例如：

```text
区间 A 价格：0.25
区间 B 价格：0.25
区间 C 价格：0.25

三者总成本：0.75
```

如果最终落入 A/B/C 中任一区间，赔付 1，盈利 0.25。
如果最终不在 A/B/C，亏损 0.75。

所以这组篮子长期赚钱的最低条件是：

```text
模型估计 A/B/C 的合计命中概率 > 75%
```

考虑滑点、手续费和安全边际后，实际要求还要更高。

### 3.2 新策略一：概率篮子交易

概率篮子交易的核心是：

```text
选择若干个区间，构造一个“命中其中任一区间就赔付 1”的篮子。
如果模型估计的覆盖概率高于市场可成交成本，则有正期望。
```

推荐优先使用等份额篮子，而不是等金额篮子。

#### 等份额篮子

假设买入三个区间：

```text
200-219：价格 0.22
220-239：价格 0.24
240-259：价格 0.18
```

如果每个区间都买 100 份：

```text
总成本 = 100 × (0.22 + 0.24 + 0.18) = 64 USDC
```

如果最终命中其中任一区间：

```text
赔付 = 100 USDC
盈利 = 100 - 64 = 36 USDC
```

这样无论命中哪个已选区间，收益都是一致的。

#### 为什么不优先等金额

如果每个区间等金额买入，不同价格会买到不同份额。

例如都买 20 USDC：

```text
0.20 价格可以买 100 份
0.40 价格只能买 50 份
```

这样最终命中不同区间时，收益差异很大，篮子结构不够干净。

### 3.3 新策略二：价格收敛交易

现实中，市场价格和真实概率通常比较接近，明显的结算正期望机会不会一直出现。

因此系统需要加入第二类策略：价格收敛交易。

价格收敛交易的目标不是持有到结算，而是在开盘或价格偏低时买入高概率区间篮子，等市场重新定价后卖出。

例如：

```text
开盘时，高概率区间篮子成本：0.68
数小时后，市场发现推文速率正常，篮子价格回升到：0.78
系统卖出，赚取 0.10 的价格差。
```

这类交易关注的是：

```text
当前价格是否偏低。
接下来几个小时，市场是否可能把这些区间重新定价。
盘口是否足够支持买入和卖出。
离结算是否还有足够时间。
```

### 3.4 两类策略的关系

推荐将交易机会分成两种标签：

```text
Settlement Edge：适合持有到中后期，模型覆盖概率明显高于成本。
Convergence Edge：适合中途卖出，目标是价格回归或市场重新定价。
```

系统界面中应该明确展示本次信号属于哪一种，不能混在一起。

## 四、入场判断公式

### 4.1 核心公式

入场判断不再只看“区间价格之和”，而是看：

```text
Edge = P_cover - Cost_cover - Fee - Slippage - Safety_margin
```

含义如下：

```text
P_cover：模型估计所选区间的总命中概率。
Cost_cover：按照真实盘口深度，买入等份额篮子的实际成本。
Fee：可能产生的交易费用。
Slippage：为了成交而支付的额外价格损耗。
Safety_margin：安全边际，用来防模型误差。
```

只有当：

```text
Edge > 0
```

才说明从结算期望角度有优势。

### 4.2 具体例子

候选区间：

```text
200-219
220-239
240-259
```

模型估计概率：

```text
200-219：28%
220-239：25%
240-259：18%
```

则：

```text
P_cover = 28% + 25% + 18% = 71%
```

盘口可成交价格：

```text
200-219：0.23
220-239：0.25
240-259：0.17
```

如果构造等份额篮子：

```text
Cost_cover = 0.23 + 0.25 + 0.17 = 65%
```

假设滑点、手续费、安全边际合计为 5%：

```text
Edge = 71% - 65% - 5% = 1%
```

这虽然是正的，但太薄，只适合观察或极小仓位。

如果市场价格变成：

```text
0.20 + 0.22 + 0.15 = 57%
```

则：

```text
Edge = 71% - 57% - 5% = 9%
```

这才是更值得考虑的机会。

### 4.3 分层入场标准

为了避免太严苛，也避免盲目交易，建议分层处理：

```text
Edge < 0：
不入场，只记录。

Edge 0%-3%：
只观察或纸面交易。

Edge 3%-8%：
小仓位试探。

Edge > 8%：
正常仓位。
```

价格收敛交易可以稍微放宽，但必须满足额外条件：

```text
高概率区间当前价格偏低。
推文速率没有恶化。
离结算还早，有时间让市场重新定价。
盘口有足够深度，后续可以退出。
市场成交量开始增加或价格正在回归。
```

### 4.4 需要注意的关键点

市场价格最高的区间，不一定最值得买。

真正要比较的是：

```text
模型概率 - 市场价格
```

或者：

```text
模型概率 / 市场价格
```

例如：

```text
区间 A：模型概率 30%，市场价格 0.29，edge 只有 1%。
区间 B：模型概率 18%，市场价格 0.11，edge 有 7%。
```

虽然 A 的概率更大，但 B 可能更便宜、更有交易价值。

## 五、概率与速率模型

### 5.1 模型不是主观判断

这里说的“模型判断”，不是让系统凭感觉猜，而是基于数据统计。

初期模型可以很朴素：

```text
历史同类周期统计。
当前 xtracker 计数。
当前时间进度。
最近速率变化。
盘口价格。
```

后期再逐步加入更复杂的统计模型。

### 5.2 第一层：历史基础概率

第一层回答的问题是：

```text
如果现在没有任何实时信息，这类周期通常会落在哪些区间？
```

需要统计：

```text
历史每日推文数量。
历史 2 天周期推文数量。
历史 7 天周期推文数量。
按星期几划分的分布。
按美国白天/夜间划分的分布。
重大事件期间和非事件期间的分布。
```

初期可以用经验分布：

```text
过去 N 个 7 天周期里，落入每个区间的频率。
```

例如：

```text
180-199：10%
200-219：28%
220-239：25%
240-259：18%
260-279：8%
其他：11%
```

这就是开盘前的 prior。

### 5.3 第二层：当前进度修正

第二层回答的问题是：

```text
现在已经过了这么久，已经发了这么多条，历史上类似情况最后通常落到哪里？
```

不要只用简单线性外推：

```text
当前数量 / 已过时间 × 总时间
```

因为马斯克发推不是稳定匀速过程，而是 bursty、事件驱动、容易阶段性爆发。

更推荐的做法是“历史相似轨迹”：

```text
当前是 7 天合约第 2 天，xtracker 计数 100。
系统查历史上第 2 天附近计数也在 90-110 的周期。
看这些周期最后分别落入哪些区间。
```

这样可以得到一个更贴近真实路径的 posterior。

简化实现：

```text
1. 把每个历史周期切成 10%、25%、50%、75% 等进度点。
2. 记录每个进度点的累计推文数。
3. 当前周期到达某个进度点时，寻找历史相似样本。
4. 统计这些样本最终落入各区间的比例。
5. 用这个比例修正开盘前的基础概率。
```

### 5.4 第三层：异常风险识别

第三层回答的问题是：

```text
现在有没有进入异常发推状态？
```

它不一定直接决定买哪个区间，但会影响是否：

```text
暂停加仓。
降低仓位。
购买尾部区间。
提前止损。
跳过本轮。
```

建议同时看几个窗口：

```text
最近 3 小时速率。
最近 6 小时速率。
最近 24 小时速率。
当前速率相对历史均值的倍数。
速率是否连续加速。
是否存在重大事件。
数据源是否同步。
```

风险状态可以分为：

```text
normal：速率正常，继续观察。
watch：轻微偏高，只提醒，不操作。
warning：明显偏高，暂停加仓，重新计算概率。
critical：严重异常，触发减仓、调仓或止损。
```

### 5.5 三层模型的最终输出

模型最终不只输出一个预测数字，而是输出一组结果：

```text
每个区间的命中概率。
所选篮子的总覆盖概率。
当前是否存在异常速率风险。
推荐操作：观察、纸面、买入、减仓、调仓、止损。
推荐理由。
```

## 六、尾部保险与 Tail Edge

### 6.1 尾部区间是什么

尾部区间是相对核心区间更低概率、更偏离当前预期的区间。

例如：

```text
核心区间：200-219、220-239、240-259
上方尾部：260-279、280-299、300+
下方尾部：160-179、180-199
```

原方案中把尾部区间称为保险仓，这个思路可以保留，但不能固定买。

### 6.2 什么是“模型尾部概率 / 市场价格有正 edge”

假设模型认为：

```text
280-299 区间真实概率：8%
市场价格：0.03
```

则：

```text
概率 / 价格 = 8% / 3% = 2.67
```

这说明市场可能低估了这个尾部风险。

这种情况下，买尾部区间不只是保险，它本身也是一笔可能有正期望的交易。

反过来：

```text
模型概率：5%
市场价格：0.12
```

则：

```text
概率 / 价格 = 5% / 12% = 0.42
```

这个尾部区间很贵，长期买会拖累收益。

### 6.3 尾部仓位的买入条件

尾部区间只有在以下条件之一满足时才买：

```text
条件一：模型概率明显高于市场价格。
条件二：当前速率进入 warning 或 critical，尾部风险正在上升。
条件三：存在重大事件，系统需要用小仓位降低极端亏损风险。
```

建议尾部仓位初期只占单轮预算的 5%-10%，不能因为恐惧而越买越多。

### 6.4 尾部仓位的角色

尾部仓位有两种角色：

```text
正期望交易：价格便宜，模型认为市场低估。
风险缓冲：价格未必便宜，但能显著降低极端情况下的组合亏损。
```

系统界面中需要明确标记每一笔尾部仓位属于哪一种。

## 七、止盈与止损体系

### 7.1 止损不能只看理论价格

原方案的单轮最大亏损 $60 是正确的纪律，但执行时要注意：

```text
不是页面显示当前价格还有 0.20，就一定能按 0.20 卖出去。
```

真实可退出价值必须按照 order book 的 bid depth 计算。

系统需要计算：

```text
如果现在立刻卖出所有持仓，按当前买盘深度能实际回收多少 USDC？
```

这个值比中间价、最新成交价更重要。

### 7.2 价格止损

价格止损是最直观的止损。

例如：

```text
单轮投入：200 USDC
最大允许亏损：60 USDC
实际可退出价值：低于 140 USDC
```

则触发价格止损。

注意这里用的是：

```text
实际可退出价值
```

不是页面显示的 mark price。

### 7.3 速率止损

速率止损指的是：

```text
虽然当前价格还没跌太多，但推文速率已经明显指向其他区间。
```

例如：

```text
你买的是 200-259 区间。
当前 xtracker 计数和最近 6 小时速率显示，最终更可能冲到 280+。
```

这时不能等市场完全反应后再卖，因为那时价格可能已经更低。

### 7.4 流动性止损

流动性止损指的是：

```text
盘口买盘变薄，继续持有可能导致后续无法退出。
```

系统要监控：

```text
当前 bid depth。
买卖价差。
如果按目标仓位卖出，会造成多大价格冲击。
```

如果流动性快速恶化，应当减仓或停止加仓。

### 7.5 时间止损

时间止损指的是：

```text
离结算越来越近，但持仓区间的胜率没有明显提高。
```

越接近结算，市场会更剧烈地重新定价。

对于 2 天合约：

```text
T+36h 前应优先清仓或大幅降低仓位。
```

对于 7 天合约：

```text
T+5 天之后，如果没有明确优势，应积极锁定利润或减少亏损。
```

### 7.6 数据异常止损

如果出现以下情况，系统应暂停新开仓，并对已有仓位进入保守模式：

```text
xtracker 无法访问。
xtracker 与备用推文 API 差异过大。
Polymarket 盘口数据异常。
订单状态无法确认。
本地数据库缺失关键数据。
```

数据异常时，系统不应该继续自动做判断。

### 7.7 组合风险止损

如果同时持有 2 天和 7 天市场，二者并不是独立风险。

同一个突发事件会同时影响两个市场。

所以需要全局止损：

```text
总敞口上限。
单日亏损上限。
单轮亏损上限。
连续亏损冷却。
跨合约联动检查。
```

## 八、半自动可视化交易驾驶舱

### 8.1 为什么先做可视化页面

直接做全自动下单风险太高。

更稳妥的路径是先做一个可视化驾驶舱，让系统把所有关键数据和判断展示出来，人来确认是否执行。

这样可以同时完成三件事：

```text
验证数据是否准确。
验证模型是否真的有 edge。
验证交易执行和风控是否可靠。
```

### 8.2 第一版页面应包含的内容

#### 市场列表

展示当前所有活跃 Elon tweets 市场：

```text
市场名称。
周期类型：2 天 / 7 天。
开始时间和结束时间。
当前 xtracker count。
距离结算剩余时间。
总成交量。
是否可交易。
```

#### 市场详情

展示每个区间 outcome：

```text
区间范围。
Yes 买一 / 卖一。
No 买一 / 卖一。
最新成交价。
盘口深度。
成交量。
模型概率。
市场价格。
edge。
```

#### 模型面板

展示：

```text
历史基础概率。
当前进度修正后的概率。
最近 3h / 6h / 24h 速率。
```

## 十一、交易流程

### 11.1 观察模式

观察模式不生成任何订单。

系统只做：

```text
拉取数据。
计算概率。
计算 edge。
记录推荐信号。
保存市场快照。
```

目标是检查数据链路是否稳定。

### 11.2 纸面交易模式

纸面交易模式生成虚拟交易记录，但不真实下单。

系统记录：

```text
如果此时按当前盘口买入，会买哪些区间。
等份额篮子的成本是多少。
模型覆盖概率是多少。
后续价格如何变化。
最终是否命中。
模拟盈亏是多少。
```

纸面交易至少运行 2-4 周。

### 11.3 半自动小额实盘

半自动阶段：

```text
系统生成信号。
用户人工确认。
系统执行限价单。
系统监控成交情况。
系统给出止盈止损建议。
关键操作仍需人工确认。
```

建议一开始只允许小仓位。

### 11.4 有限自动化

当纸面交易和半自动交易都验证有效后，可以逐步开启：

```text
自动撤销长时间未成交订单。
自动执行硬止损。
自动执行已确认规则下的部分止盈。
自动暂停异常市场。
```

开仓仍建议长期保留人工确认，直到策略有足够样本。

## 十二、入场过滤器 V2

### 12.1 必要条件

系统只有在以下条件全部满足时才允许生成真实订单：

```text
xtracker 数据可用。
Polymarket 盘口数据可用。
目标 outcome 有足够 ask depth。
选中篮子有明确策略标签：Settlement Edge 或 Convergence Edge。
总敞口未超限。
当前没有触发数据异常。
用户处于允许交易模式。
```

### 12.2 Settlement Edge 入场

适合持有到中后期的概率篮子交易。

必要条件：

```text
P_cover > Cost_cover + Fee + Slippage + Safety_margin
Edge 至少大于 3%。
最好大于 8% 才正常仓位。
```

### 12.3 Convergence Edge 入场

适合中途卖出的价格收敛交易。

必要条件：

```text
高概率区间篮子当前价格低于历史合理区间。
当前推文速率 normal 或 watch。
距离结算仍有足够时间。
盘口流动性支持后续退出。
近期成交量或价格变化显示市场可能重新定价。
```

这类交易的仓位应该小于 Settlement Edge。

### 12.4 不入场条件

以下情况直接跳过：

```text
模型概率和市场价格非常接近，没有明显优势。
盘口太薄。
买卖价差太大。
接近结算但没有明确命中优势。
数据源不同步。
近期已经连续亏损。
同一事件导致多个市场高度相关，组合风险过高。
```

## 十三、仓位与资金管理

### 13.1 初始资金假设

沿用原方案的示例：

```text
总资金：500 USDC
最大总敞口：400 USDC
全局预留：100 USDC
单轮预算：200 USDC
单轮最大亏损：60 USDC
```

### 13.2 推荐仓位分层

根据 edge 分层：

```text
观察 / 纸面：0 USDC
Edge 0%-3%：0 或极小纸面仓位
Edge 3%-8%：正常单轮预算的 25%-50%
Edge > 8%：正常单轮预算的 50%-100%
```

价格收敛交易默认降低仓位：

```text
Convergence Edge 仓位不超过 Settlement Edge 仓位的 50%-70%。
```

### 13.3 核心仓位

核心仓位优先买等份额篮子。

例如单轮预算 200 USDC，系统计算后决定核心篮子最大成本为 120 USDC。

如果选中三个区间，价格分别为：

```text
0.22、0.24、0.18
```

单份篮子成本为：

```text
0.64
```

可买份额约为：

```text
120 / 0.64 = 187.5 份
```

实际下单时还要根据 tick size、最小订单规模、盘口深度做调整。

### 13.4 预留资金

预留资金不应该默认全部用掉。

用途包括：

```text
第二阶段加仓。
异常时向新概率区间平移。
尾部风险突然变便宜时买入。
保留流动性，避免被动。
```

### 13.5 并行市场

如果同时持有 2 天和 7 天市场：

```text
总敞口降低。
单个市场仓位降低。
一个市场触发速率止损时，另一个市场必须重新评估。
```

## 十四、订单执行设计

### 14.1 默认使用限价单

系统默认生成限价单，不使用无保护的市价化追单。

每笔订单需要记录：

```text
目标 token_id。
买入或卖出。
限价价格。
份额数量。
对应篮子编号。
下单理由。
风控状态。
```

### 14.2 部分成交处理

部分成交是这个策略的关键风险。

如果篮子只成交了一部分，就可能变成裸露方向仓位。

因此系统需要：

```text
下单后立即监控成交情况。
在指定时间内检查是否完整成交。
如果只成交部分，自动提示用户补齐、撤单或反向平仓。
不允许长时间处于不完整篮子状态。
```

### 14.3 订单草稿

半自动阶段先生成订单草稿：

```text
用户看到订单列表。
用户看到预期成本、最大亏损、命中收益。
用户确认后才提交。
```

这能避免模型直接交易。

## 十八、最终结论

这个系统应该先做成一个半自动化的可视化交易驾驶舱，而不是一开始就做全自动交易机器人。

新的核心原则是：

```text
先看得见，再算得清，再小额验证，最后才自动化。
```

策略上不再把“几个区间价格和小于 100%”当成无风险套利，而是明确分为：

```text
概率篮子交易：模型覆盖概率高于真实可成交成本。
价格收敛交易：市场短期低估高概率区间，等待重新定价后卖出。
```

产品上先建设：

```text
数据采集。
概率模型。
篮子模拟器。
纸面交易。
人工确认下单。
风控告警。
复盘系统。
```

等系统积累足够样本，证明模型 edge 和价格收敛信号确实有效，再逐步开放自动撤单、自动止损、自动止盈，最后再讨论是否全自动开仓。
